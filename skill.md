# Serverless GitHub-Native SPA Architecture

**Use this skill when you are:**
- Building a tool that doesn't need a backend server (web-first, offline-capable, read-heavy)
- Designing infrastructure resilient to vendor/org disappearance (decentralized, self-hostable)
- Creating a service anyone can fork/mirror without infrastructure costs
- Publishing data generated by GitHub Actions (batch jobs, cron analysis, metadata extraction)
- Accepting user submissions without requiring accounts or tokens
- Building open-source infrastructure independent of central organization

## Philosophy

**GitHub as Complete Infrastructure:**
- **Hosting:** GitHub Pages (free, CDN, static)
- **Processing:** GitHub Actions (free, scheduled, event-triggered)
- **State:** JSON in Git (version-controlled, distributed, auditable)
- **Submissions:** User \E2\86\92 Issue \E2\86\92 Bot reads issue \E2\86\92 Bot creates PR (no auth needed)

**Benefits:**
- Zero server cost or operational burden
- Service survives org/foundation closure (fork, mirror, run anywhere)
- Anyone can run locally (clone \E2\86\92 Python script \E2\86\92 serve)
- Transparent: all logic in Git, all state in PRs, audit trail visible
- Decentralized: no authentication required from end users, no account overhead

## Core Components

### 1. Data Generation Script (`scripts/generate_data.py`)

```python
#!/usr/bin/env python3
"""Generate JSON data from source repos or APIs."""
import json
import subprocess
from pathlib import Path

def generate_data():
    """
    Run batch processing and output JSON to data/.
    Called by GitHub Actions on schedule or webhook.
    """
    # 1. Fetch/clone sources
    sources = fetch_sources()  # Git, API, filesystem

    # 2. Process data (regex, AST, custom analysis)
    processed = [process_item(item) for item in sources]

    # 3. Write JSON files
    Path("data").mkdir(exist_ok=True)

    # Per-item detail
    for item in processed:
        path = Path("data") / f"{item['id']}.json"
        path.write_text(json.dumps(item, indent=2))

    # Aggregated indices
    (Path("data") / "index.json").write_text(
        json.dumps({"items": [x["id"] for x in processed]}, indent=2)
    )
    (Path("data") / "stats.json").write_text(
        json.dumps(compute_stats(processed), indent=2)
    )

if __name__ == "__main__":
    generate_data()
```

**Characteristics:**
- Runs on **schedule** (daily via Actions) or **on-demand** (workflow_dispatch)
- **Offline analysis** \E2\80\94 clones once, processes locally
- **Outputs JSON** to `data/` (committed to Git, versioned)
- Deterministic, repeatable, **no external state**

### 2. GitHub Actions Workflows

#### `update_data.yml` \E2\80\94 Scheduled Data Refresh

```yaml
name: Update Data
on:
  schedule:
    - cron: '0 2 * * *'  # Daily at 2am UTC
  workflow_dispatch:  # Manual trigger

jobs:
  generate:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v4
        with:
          python-version: '3.10'
      - run: pip install -e ".[dev]"
      - run: python scripts/generate_data.py
      - uses: stefanzweifel/git-auto-commit-action@v5
        with:
          commit_message: 'chore: regenerate data'
          file_pattern: 'data/*'
```

**Pattern:** Run Python script \E2\86\92 auto-commit diffs to Git \E2\86\92 Pages automatically re-deployed.

#### `process_submission.yml` \E2\80\94 Issue-to-PR Bot

User opens issue with JSON payload. Bot reads, validates, creates PR.

```yaml
name: Process Submission
on:
  issues:
    types: [opened]

jobs:
  process:
    # Only run if issue title matches marker (e.g., "[submission]")
    if: startsWith(github.event.issue.title, '[submission]')
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Parse issue body
        id: parse
        run: |
          BODY="${{ github.event.issue.body }}"
          # Extract JSON from code fence in issue body
          JSON=$(echo "$BODY" | sed -n '/```json/,/```/p' | sed '1d;$d')

          # Validate JSON
          echo "$JSON" | python -m json.tool > /dev/null
          echo "payload=$JSON" >> $GITHUB_OUTPUT

      - name: Create PR from submission
        env:
          GH_TOKEN: ${{ secrets.GITHUB_TOKEN }}
        run: |
          BRANCH="submission-${{ github.event.issue.number }}"
          PAYLOAD='${{ steps.parse.outputs.payload }}'

          git checkout -b "$BRANCH"

          # Apply submission (e.g., write files)
          echo "$PAYLOAD" | python scripts/apply_submission.py

          git config user.name "github-actions[bot]"
          git config user.email "github-actions[bot]@users.noreply.github.com"
          git add .
          git commit -m "feat: process submission #${{ github.event.issue.number }}"
          git push origin "$BRANCH"

          # Create PR
          gh pr create \
            --title "submission: process issue #${{ github.event.issue.number }}" \
            --body "Generated from submission: ${{ github.event.issue.html_url }}" \
            --base main
```

**Why this pattern (no user auth):**
- User opens issue via GitHub web (no token needed, everyone has account for read-only access)
- Bot reads issue (uses repo's `GITHUB_TOKEN`, always available in Actions)
- Bot creates PR (from Actions, transparent, logged, reviewable)
- No need to ask user for PAT or OAuth flow

### 3. SPA (`index.html`)

**Architecture:**
- Single HTML file (no build step, no toolchain)
- Fetch `data/*.json` via HTTP (from Pages)
- Vanilla JavaScript + **Tailwind CSS** (CDN)
- Submit action \E2\86\92 open issue on GitHub (link in SPA)

**Key sections:**
```html
<header>  <!-- Nav, title, settings -->
<main>
  <aside>  <!-- Data browser/filters -->
  <section>  <!-- Detail view + editor -->
</main>
<script>
  // Load data
  async function loadData() {
    const resp = await fetch('/data/index.json');
    return resp.json();
  }

  // Submit \E2\86\92 generate issue body
  function prepareSubmission() {
    const payload = { /* user edits */ };
    const issueBody = `
[submission] Update data

\`\`\`json
${JSON.stringify(payload, null, 2)}
\`\`\`

Submitted via SPA.
`;
    return issueBody;
  }

  // Guide user to open issue
  function showSubmitDialog() {
    const body = prepareSubmission();
    const url = new URL('https://github.com/owner/repo/issues/new');
    url.searchParams.set('title', '[submission] Update');
    url.searchParams.set('body', body);
    window.open(url.href);
  }
</script>
```

**Submit flow (user perspective):**
1. Make edits in SPA
2. Click "Submit" button
3. SPA opens GitHub issue form (pre-filled with JSON)
4. User reviews, adds comments, submits issue
5. Bot reads issue, creates PR
6. PR waits for human review before merge

## Scaling & Resilience

**Vendor Independence:**
- Service doesn't depend on any org or domain
- If current owner goes away: anyone forks, enables Pages, runs on own domain
- No infrastructure to migrate (it's all in Git + GitHub Actions, both free)

**Mirror Setup (for any reason):**
```bash
# Clone to new org/account
git clone https://github.com/original-owner/my-spa my-spa-mirror
cd my-spa-mirror
git remote set-url origin https://github.com/new-owner/my-spa-mirror

# Enable GitHub Pages on new location
# Adjust .github/workflows if needed (e.g., which repos to scan)
git push origin main
```

**Example: Community Fork**
If original service is unmaintained, community can:
1. Fork repo
2. Push to alternate org
3. GitHub Pages auto-serves from new domain
4. Same bot workflows, same data, same SPA
5. No code changes needed

## When to Use This Pattern

\E2\9C\85 **Good fit:**
- Dashboards, analytics, data browsers
- Metadata aggregation (indexes, catalogs, reports)
- Community contribution workflows (via issues + bot PRs)
- Archival/documentation sites
- Tools that should survive org closure
- Services with <5min latency tolerance
- Data <100MB total

\E2\9D\8C **Not a fit:**
- Real-time systems (<1min sync)
- Large datasets (>500MB)
- User accounts/auth (no backend identity)
- High-write-frequency (constant API updates)
- Complex transactions

## Key Principles

1. **Serverless** \E2\80\94 GitHub Pages + Actions, no EC2/Lambda/containers
2. **Offline-First** \E2\80\94 SPA works without network after load
3. **Data as Code** \E2\80\94 All state in Git, all logic in Python/JavaScript
4. **No User Auth** \E2\80\94 Issues don't require login, bot uses repo token
5. **Distributed** \E2\80\94 Forkable, mirrored, org-independent
6. **Transparent** \E2\80\94 Full audit trail, PRs reviewed before merge
7. **Zero Cost** \E2\80\94 Free tier sufficient for most use cases
